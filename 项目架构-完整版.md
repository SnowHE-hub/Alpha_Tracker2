# Alpha_Tracker2 项目架构（完整版）

> 若将**价量规则 + 单标的/截面 ML + 深度学习 + 基本面 + 研报/新闻/公告 + 行业板块 + 主力/北向/换手**等全部做出来，整体架构与模块应如何设计？本文档给出**分层架构、模块清单、目录结构、接口约定与数据流**，与《设计文档-完整版》《选股模型设计》《数据工程》《扩展模型与数据工程》一致，作为实施蓝图。

---

## 0. 架构总览

- **分层**：数据层 → 特征层 → 模型层 → 策略层 → 组合与执行层 → 评估层 → 报告与应用层；训练与实验独立成块。
- **原则**：配置统一、存储统一（DuckDB + lake）、版本/策略隔离、插件化可插拔。
- **产出**：同一套行情与规则下，多模型/多策略的 signals → picks → trades → NAV → 评估指标 → Dashboard/报表。

## 4. 接口约定（关键边界）

### 4.1 数据 → 特征

- **Ingestion** 只负责：调用外部 API、写 DuckDB 表 + 写 lake（Parquet）；不做衍生。
- **build_features / build_features_extended**：只读 store（及必要时的 lake），写 features_daily（或扩展表）；不读外部 API。

### 4.2 特征 → 模型

- **Scorer**：`score(trade_date, features: DataFrame) -> DataFrame`；features 至少含 ticker，其余列按版本要求；输出含 ticker, score, score_100, reason, …
- **score_all**：统一加载 universe、features，按 config 的 score_versions 循环调用 scorer，再写 picks_daily、做阈值/fallback；scorer 不写库。

### 4.3 模型 → 策略 → 组合

- **Strategy**（若独立）：`build_signals(predictions, universe, date)`、`build_target_portfolio(signals, state, date)`；当前部分逻辑在 score_all 与 generate_orders 内，可逐步抽到 strategies/plugins。
- **组合/执行**：读 picks_daily 或 signals；输出 orders → fills → positions_daily, trades_daily, nav_daily。

### 4.4 配置

- 所有路径、数据源开关、模型 version 与 model_path、阈值参数，均来自 **configs/default.yaml**（或 experiments/*.yaml）；代码不写死。

### 4.5 存储

- 仅 **DuckDBStore** 写 DuckDB；所有表结构在 **storage/schema.sql** 中定义；大面板优先写 **lake**，再按需入 store。

## 7. 小结

- **架构**：数据层 → 特征层 → 模型层 → 策略层 → 组合与执行层 → 评估层 → 报告与应用层；训练与实验独立；配置与存储统一。
- **模块**：按上表逐层落实；Ingestion 按数据类型拆（universe, prices, fundamentals, sectors, fund_flow, events, lhb）；Features 拆价量与扩展；Scoring 按模型类型拆插件；Strategies/Portfolio/Execution 可从现有 pipeline 逐步抽离。
- **目录**：在现有 src/alpha_tracker2 下增加 features/、training/、evaluation/、reporting/（可选），扩展 ingestion/plugins、scoring/plugins、pipelines；data/lake 分子目录，data/models 存模型文件。
- **接口**：Scorer、Strategy、Provider 均以 Protocol/registry 约定；配置与 DuckDB 契约与现有设计文档一致。按 P0→P1→…→P5 推进即可在现有项目上平滑扩展至“全量数据+多模型+多策略”的完整架构。
