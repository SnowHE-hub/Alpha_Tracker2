# Alpha_Tracker2 选股模型设计（V1–V5 重设计）

> 本文档为**选股层（Scorer / Model Plugin）**的完整设计思路，用于在保持现有 pipeline 与契约的前提下，重新定义 V1–V4 的职责与公式，并预留 V5。实现时需与 RUNBOOK、设计文档-完整版 一致。

---

## 0. 设计目标与原则

### 0.1 目标

- **版本间职责清晰**：每个版本有明确的“人设”（baseline / 进攻 / 稳健 / 趋势质量 / 截面模型），便于对比与迭代。
- **可配置**：权重、边界、阈值按版本可调，避免写死。
- **可解释**：所有版本输出结构化 reason（JSON 或兼容字段），便于归因与 Dashboard。
- **可扩展**：新增 V5 或调整某版本不破坏现有契约（picks_daily、score_all、阈值机制）。

### 0.2 原则

- **Scorer 只负责打分**：是否入选、fallback 仍由 score_all 统一处理；阈值仍来自 ab_threshold_history.json，按 version 分桶。
- **兼容现有 pipeline**：输入仍为 features_daily + trade_date；输出仍为含 ticker, score, score_100, reason, picked_by 等列的 DataFrame；V2/V3/V4 继续走“阈值 + fallback_topk”。
- **配置优先**：新增参数一律进 configs/default.yaml（或 per-version 节点），代码只读配置。

---

## 1. 版本总览与定位

| 版本 | 定位（人设） | 核心思路 | 是否用 ML | 是否用阈值 | 主要差异点 |
|------|----------------|----------|-----------|------------|------------|
| **V1** | 纯规则基线 | 多因子 z-score 加权，无 ML、无阈值 | 否 | 否 | 可解释性最强；作为对比锚点 |
| **V2** | 趋势 + ML（偏进攻） | 趋势分 + 单标的 ML 概率 + 预期收益 + 风险惩罚 | 是 | 是 | 风险权重适中，追求收益 |
| **V3** | 趋势 + ML（偏稳健） | 同 V2 骨架，提高风险权重、降低趋势权重 | 是 | 是（更严） | 同公式不同权重 + 更高 q |
| **V4** | 趋势质量 + 回测验证 | V2 骨架 + 滚动回测得分(bt_*) + MA 对齐加成 | 是 | 是 | 用 features 里 bt_*，强调“历史能赚钱的形态” |
| **V5** | 截面/池化模型（可选） | 全市场一个模型，截面特征 + 时间序列 | 是 | 是 | 与 V2–V4 单标的 ML 形成对照 |

## 2. V1：纯规则基线（Rule Baseline）

### 2.1 角色

- 不做任何 ML、不做阈值筛选；**全 universe 按分数排序**，选股完全由下游（如 TopK 或策略层）决定。
- 作为**可解释、可复现的对比锚点**：任何 V2/V3/V4 的改进都应相对 V1 有稳定优势（收益/夏普/命中率等）。

### 2.2 输入

- **必选特征**：`ticker`, `ret_1d`, `mom_5d`, `vol_5d`（与现有一致）。
- **可选扩展**：`ret_5d`, `ret_10d`（若存在则参与加权，权重可配）；`avg_amount_20`（若存在可用于软过滤或降权，不硬删）。

### 2.3 公式

- 对每个因子做**截面 z-score**（与现有一致）；缺失或 std=0 时 z 取 0。
- **加权和**：
  - `score = w_mom * z(mom_5d) + w_vol * (-z(vol_5d)) + w_ret1 * z(ret_1d) + w_ret5 * z(ret_5d) + w_ret10 * z(ret_10d)`
  - 其中 `ret_5d` / `ret_10d` 若不存在则对应项为 0；权重来自配置，默认与当前一致：`w_mom=0.6, w_vol=0.3, w_ret1=0.1`，扩展时例如 `w_ret5=0.05, w_ret10=0.05` 并归一化。
- **可选软过滤**：对 `avg_amount_20` 低于某分位数（如 10%）的标的，可在 score 上减一个常数（如 -0.5）而不是直接剔除，保证 universe 不变、可解释。

### 2.4 输出

- 列：`ticker`, `name`, `rank`, `score`, `reason`, `picked_by`, `trade_date`。
- **score**：原始加权 z 和（可再线性映射到 0–100 作为 score_100，由 score_all 统一做）。
- **reason**：JSON，包含：
  - `method`: 简短描述（如 "V1 rule: 0.6*z(mom_5d)+0.3*(-z(vol_5d))+0.1*z(ret_1d)"）；
  - `signals`: 各因子原始值；
  - `z_scores`: 各因子 z 值；
  - `weights`: 各因子权重；
  - `contrib`: 各因子贡献（weight * z）；
  - `score_raw`。
- **picked_by**：固定 `"BASELINE_RANK"`（由 score_all 写，scorer 只保证不写 THRESHOLD/FALLBACK）。

### 2.5 配置建议（default.yaml）

```yaml
scoring:
  v1:
    weights:
      mom_5d: 0.60
      vol_5d: 0.30   # 实际用 -z(vol)
      ret_1d: 0.10
      ret_5d: 0.0    # 可选
      ret_10d: 0.0   # 可选
    # 可选：软过滤流动性
    liquidity_soft_filter:
      enabled: false
      amount_pct_min: 0.10   # 低于 10% 分位时扣分
      score_penalty: 0.5
```

## 3. V2：趋势 + ML（偏进攻，Aggressive）

### 3.1 角色

- 在**趋势质量 + 单标的 ML 预测**基础上，追求收益；风险惩罚存在但不主导。
- 与 V3 共用同一套“趋势 + ML + 预期收益 + 风险惩罚”骨架，**通过权重区分**：V2 趋势与概率权重大、风险权重适中。

### 3.2 输入

- **features_daily**：含 trend 相关（ret_10d, ret_20d, ma5_gt_ma10_gt_ma20, ma20_above_ma60, ma20_slope）、风险相关（vol_ann_60d, mdd_60d, limit_down_60）、ticker, trade_date 等。
- **价格序列**：从 prices_daily 拉取单标的历史，用于单标的 LR+GBDT 预测 5 日上涨概率（与现有一致）。

### 3.3 公式（与现有 v23 对齐，参数可配置）

- **趋势分 trend** [0,1]：  
  - 收益项：ret_10d、ret_20d、ma20_slope 线性映射到 0–1（worst/best 可配置）。  
  - 均线项：ma5>ma10>ma20、ma20>ma60 给 1.0/0.3/0.4 等离散分。  
  - 惩罚项：vol_ann_60d、mdd_60d、limit_down_60 越高惩罚越大。  
  - `trend = raw_trend * (1 - penalty)`，再 clamp 到 [0,1]。
- **ML 概率 prob_ens** [0,1]：单标的 LR+GBDT 对 5 日涨跌的预测概率；缺失时 0.5。
- **预期收益 exp_ret**：`prob_ens * ret10`，再线性映射到 0–1（worst/best 可配置）。
- **风险惩罚 penalty** [0,1]：与 trend 内惩罚一致或单独算一版，映射到 0–1。
- **最终**：  
  `final01 = W_TREND * trend + W_PROB * prob_ens + W_EXPECT * exp_score + W_RISK * (1 - penalty)`  
  **V2 默认权重**：W_TREND=0.35, W_PROB=0.45, W_EXPECT=0.10, W_RISK=0.10。  
  `score_100 = final01 * 100`。

### 3.4 输出

- 列：`ticker`, `name`, `score`, `score_100`, `reason`, …（trade_date 等由 score_all 补）。
- **reason**：结构化 JSON，至少包含：
  - `trend`, `prob_ens`, `exp_ret`, `penalty`, `final01`, `method`: "V2 aggressive: trend+ML+exp+risk"。

### 3.5 阈值与选股

- 由 score_all 使用 **V2** 的 ab_threshold_history 做滚动分位数阈值；fallback_topk 使用配置的 topk_fallback。
- 配置支持**按版本**的 q/window，例如 V2 用 `q=0.80, window=60`。

## 4. V3：趋势 + ML（偏稳健，Conservative）

### 4.1 角色

- **同骨架、不同权重**：更重视风险、略降趋势与概率权重，目标“少而稳”的标的。
- 与 V2 的差异**不仅**是阈值历史，而是**公式内权重** + **更严阈值**（更高 q 或更长 window）。

### 4.2 公式

- 趋势/ML/预期收益/风险惩罚的**计算方式**与 V2 完全一致（复用同一套 trend、prob_ens、exp_ret、penalty）。
- **V3 专用权重**（建议）：  
  `W_TREND=0.25, W_PROB=0.35, W_EXPECT=0.10, W_RISK=0.30`  
  即风险权重提高到 30%，趋势与概率相应降低。
- **最终**：`final01 = ...` 同式，仅权重不同；`score_100 = final01 * 100`。

### 4.3 阈值与选股

- score_all 使用 **V3** 的 ab_threshold_history；**建议 V3 使用更严参数**：例如 `q=0.88, window=60` 或 `q=0.85, window=90`，在配置中与 V2 分开。

### 4.4 输出与配置

- 与 V2 相同的输出列与 reason 结构；`method` 标明 "V3 conservative: trend+ML+exp+risk (high risk weight)"。
- 配置中为 V3 单独设 `weights` 与 `threshold: { q, window }`。

## 5. V4：趋势质量 + 回测验证（Trend Quality + Backtest）

### 5.1 角色

- 在 V2 的“趋势 + ML + 风险”基础上，**显式加入“历史滚动回测表现”**（build_features 已算的 bt_mean、bt_winrate 等）和**均线多头排列加成**。
- 目标：选出“不仅当下趋势好、ML 看多，而且**过去同样形态下持有 N 日表现也好**”的标的。

### 5.2 输入

- 除 V2/V3 所需外，**必须**使用 features_daily 中的：`bt_mean`, `bt_winrate`（或 `bt_median`）；可选 `bt_best_style`。
- 均线：`ma5_gt_ma10_gt_ma20`（已有）。

### 5.3 公式

- **V2 核心**：与 V2 相同算得 `v2_core01`（即 trend + prob + exp + risk 的 final01）。
- **回测得分 bt_score** [0,1]：  
  - 由 `bt_mean`、`bt_winrate` 线性映射到 0–1（worst/best 可配置）；若缺失则 bt_score=0。  
  - 可选：`bt_score = 0.5 * score01(bt_mean) + 0.5 * score01(bt_winrate)`。
- **MA 加成 ma_bonus** [0, 0.02]：  
  - 当 `ma5_gt_ma10_gt_ma20 == 1` 时取 `ma_bonus_value`（默认 0.02），否则 0。
- **V4 最终**：  
  `v4_final01 = clamp( W_CORE * v2_core01 + W_BT * bt_score + ma_bonus, 0, 1 )`  
  建议默认：W_CORE=0.78, W_BT=0.20, ma_bonus_value=0.02（即约 78% 核心 + 20% 回测 + 2% MA 加成）。

### 5.4 输出

- 列同 V2/V3；**reason** 中增加：`v2_core01`, `bt_mean`, `bt_winrate`, `bt_score`, `ma_bonus`, `method`: "V4 trend+ML+bt+MA"。

### 5.5 阈值与配置

- 阈值仍用 **V4** 的 ab_threshold_history；q/window 可与 V2 相同或略松（例如 q=0.78）。
- 配置中为 V4 单独设：`w_core`, `w_bt`, `ma_bonus_value`，以及 trend 的 worst/best（若与 V2/V3 共用则放 common）。

## 6. V5：截面/池化模型（可选，Phase 2）

### 6.1 角色

- **全市场一个模型**：用同一套截面特征（+ 可选时间序列摘要）对所有标的打分，利用“截面相对强弱”与潜在行业/风格信息。
- 与 V2/V3/V4 的“单标的 ML”形成对照，便于评估“截面信息是否有边际”。

### 6.2 思路（不实现细节，仅设计）

- **特征**：features_daily 的数值列（ret_*、vol_*、ma_*、bt_* 等）+ 可选行业/市值分箱；同一 trade_date 一条记录 per ticker，组成面板。
- **标签**：5 日后是否上涨（或 5 日收益），需用未来价格构造，**仅用于训练**；预测时无未来数据。
- **模型**：例如 LogisticRegression / LightGBM / 小型 MLP；训练时用 T-1 日及之前数据，预测 T 日截面。
- **输出**：每个 ticker 一个 score（或 prob_up_5d）；可再与简单规则（如波动过滤）结合。
- **校准**：Platt 或 isotonic，使预测概率与实盘命中率接近。
- **阈值**：与 V2/V3/V4 相同机制，使用 **V5** 的 ab_threshold_history。

### 6.3 与现有 pipeline 的衔接

- 新增 scorer 插件 `V5PooledScorer`；在 registry 与 default.yaml 的 score_versions 中加入 V5。
- 训练可在离线脚本中完成，每日只跑“加载模型 + 预测”；模型文件路径与版本管理由配置指定。

## 7. 配置结构总览（default.yaml）

```yaml
scoring:
  score_versions: ["V1", "V2", "V3", "V4"]   # 可选加 "V5"

  # V1：纯规则
  v1:
    weights:
      mom_5d: 0.60
      vol_5d: 0.30
      ret_1d: 0.10
      ret_5d: 0.0
      ret_10d: 0.0
    liquidity_soft_filter:
      enabled: false
      amount_pct_min: 0.10
      score_penalty: 0.5

  # V2/V3/V4 共用：趋势分边界、ML 参数、阈值默认
  trend_bounds:          # 用于 trend 与 penalty 的 worst/best
    ret_10d: [-0.05, 0.12]
    ret_20d: [-0.08, 0.20]
    ma20_slope: [-0.003, 0.006]
    vol_ann_60d: [0.25, 0.60]
    mdd_60d: [0.06, 0.18]
  ml:
    lookback_days: 260
    min_train_samples: 80
    pred_horizon_days: 5

  # 按版本：权重 + 阈值
  versions:
    V2:
      weights:
        trend: 0.35
        prob: 0.45
        expect: 0.10
        risk: 0.10
      threshold:
        q: 0.80
        window: 60
      fallback_topk: 50
    V3:
      weights:
        trend: 0.25
        prob: 0.35
        expect: 0.10
        risk: 0.30
      threshold:
        q: 0.88
        window: 60
      fallback_topk: 30
    V4:
      w_core: 0.78
      w_bt: 0.20
      ma_bonus_value: 0.02
      bt_bounds:
        bt_mean: [-0.05, 0.15]
        bt_winrate: [0.35, 0.65]
      threshold:
        q: 0.78
        window: 60
      fallback_topk: 50

  # 兼容旧 key（若保留）
  v2_v3_v4:
    common:
      q: 0.80
      window: 60
      topk_fallback: 50
```

- 若采用“按版本 versions.V2/V3/V4”结构，score_all 与 thresholds 需改为从 `scoring.versions[version]` 读 q/window/fallback_topk；否则可暂时仍用 common，仅 V3 用不同权重、V4 用 w_core/w_bt/ma_bonus。

## 8. 契约与兼容性

### 8.1 picks_daily 契约（不变）

- 主键：`(trade_date, version, ticker)`。
- 字段：score, score_100, thr_value, pass_thr, picked_by, reason, … 语义与现有 RUNBOOK 一致。
- Scorer 不决定“是否入选”，只输出 score；score_all 负责 thr、pass_thr、fallback、picked_by。

### 8.2 阈值机制（不变）

- 阈值来源：ab_threshold_history.json，按 version 分桶。
- 更新基于**全量候选 score**（当日 universe），禁止用“已筛选后样本”更新。
- 各版本 q/window 建议从配置读取，支持 per-version。

### 8.3 reason 结构（增强）

- **V1**：已为 JSON；保持并可选增加 ret_5d/ret_10d 的 contrib。
- **V2/V3/V4**：当前为短字符串；**重设计后**统一为 JSON，字段至少包含：  
  `trend`, `prob_ens`, `exp_ret`, `penalty`, `final01`, `method`；V4 额外：`v2_core01`, `bt_score`, `ma_bonus`。
- Dashboard 与评估脚本若已解析 reason，需兼容“旧字符串 + 新 JSON”或仅解析新 JSON。

## 9. 实现顺序建议

| 阶段 | 内容 | 说明 |
|------|------|------|
| **1** | 配置与权重可配置化 | 将 V1 权重、V2/V3 的 W_*、趋势 worst/best 移入 default.yaml；V2/V3 从配置读权重，V3 使用与 V2 不同的默认权重。 |
| **2** | V3 公式区分 | 在 _score_row_v23 或调用处根据 version 选择权重集（V2 vs V3）；保证 V2/V3 输出列与 reason 结构一致。 |
| **3** | V4 接入 bt_* | 在 V4 scorer 中读取 bt_mean、bt_winrate，计算 bt_score，与 v2_core01、ma_bonus 组合；ma_bonus 从配置读取。 |
| **4** | V1 扩展与软过滤 | V1 支持 ret_5d/ret_10d 权重（可选）；可选流动性软过滤；reason 保持 JSON。 |
| **5** | V2/V3/V4 reason 结构化 | 将 reason 从短字符串改为 JSON，包含 trend/prob_ens/exp_ret/penalty/final01/method 等。 |
| **6** | 按版本阈值配置 | score_all 与 thresholds 支持从 scoring.versions[version].threshold 读 q/window/fallback_topk。 |
| **7** | V5（可选） | 单独分支：特征工程、训练脚本、V5PooledScorer、模型路径配置、registry 注册。 |

## 10. 小结

- **V1**：纯规则、多因子 z-score、可配置权重与可选软过滤；无 ML、无阈值；全量排序，作为基线。
- **V2**：趋势 + 单标的 ML + 预期收益 + 风险惩罚；权重偏进攻；阈值 + fallback。
- **V3**：同骨架、权重偏稳健（高风险权重）+ 更严阈值；与 V2 明确区分。
- **V4**：V2 核心 + 滚动回测得分(bt_*) + MA 加成；强调“趋势质量与历史表现”。
- **V5**：可选截面/池化模型；与单标的 ML 对照，便于后续扩展。

所有改动保持**契约不变、配置驱动、可解释性增强**；实现时按阶段 1–6 顺序推进，V5 视需求再开分支。
