Step 2：统一“日期语义”与“评估口径”（trade_date / asof / horizon）

这是你反复强调的坑：信号日≠评估日。你设计思路里也把它列为核心工程约束。 

设计思路

目标：所有评估相关 pipeline（eval_5d / batch / dashboard 展示）使用同一套约定：

trade_date：产生信号/打分的日期

asof：站在某天回看（walk-forward）

horizon：未来 1/5/10（后面扩展）

picks_date = asof - horizon（你旧文档里也强调过）

完成定义：

dashboard 上展示指标时，用户不会再误解“今天的 picks 用今天的未来收益评估”；

batch 输出能清楚标注“评估窗口”。

你设计思路里写得很清楚：

Strategy Plugin：prediction → target portfolio（TopK/AB/权重/调仓）

Execution & Tracking：orders → fills → state → NAV → drawdown 

设计思路

你旧的 portfolio_live_tracker.py 本质已经是 Execution & Tracking 的雏形（state、trades、nav、drawdown、positions_history、signals_archive、5日验证）。 

设计思路

目标：不动旧 tracker 的正确性前提下，把它的能力拆成：

risk rules（止损/止盈/最大持有期）

rebalancer（等权/补齐/换仓）

broker_sim（成交价模型、滑点、费率）

state store（每策略隔离的账户状态）

完成定义：

以后你要加“基金/期货”时，只换 asset rules + fees + execution，不用推倒选股层。
