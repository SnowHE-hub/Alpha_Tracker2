# Alpha_Tracker2 数据工程：模型数据需求与方案

> 本文档说明构建 V1–V5 选股模型**需要从 akshare 及其他渠道获取哪些数据**、**如何获取**，以及**数据工程应如何组织**（分层、缓存、回填、质量）。

---

## 0. 总览

| 数据类型 | 当前来源 | 用途（模型） | 是否必须 | 备注 |
|----------|----------|----------------|----------|------|
| 股票列表（代码+名称） | akshare | Universe、所有版本 | 是 | 已有 |
| 日频行情（OHLCV+成交额） | akshare | 全量特征、V1–V4、V5 特征 | 是 | 已有 |
| 交易日历 | akshare | 窗口计算、评估 horizon | 是 | 已有 |
| 行业/板块 | akshare（可选） | V5 截面、行业分散、分析 | 可选 | 可增强 |
| 市值/流通市值 | akshare 或 其他 | V5、流动性过滤、分层 | 可选 | 可增强 |
| 基本面（PE/PB等） | 其他渠道 | 未来扩展 | 否 | 预留 |
| 复权方式 | akshare 参数 | 收益一致性 | 建议 | 当前未复权 |

## 1. 各模型依赖的数据（逐层拆解）

### 1.1 数据流简图

```
[akshare / 其他 API]
        │
        ▼
  ingest_universe ──► picks_daily(version=UNIVERSE) + lake/universe/
  ingest_prices   ──► prices_daily + lake/prices/
        │
        ▼
  build_features  ──► features_daily（只用 prices_daily 的 close + amount）
        │
        ▼
  score_all (V1–V4)  用 features_daily + (V2/V3/V4 再读 prices_daily 做单标的 ML)
  V5（若上）         用 features_daily + 可选行业/市值表
```

### 1.2 V1（纯规则）

| 需求 | 来源 | 说明 |
|------|------|------|
| ret_1d, mom_5d, vol_5d | features_daily | 由 build_features 从 **close** 算出 |
| 可选 ret_5d, ret_10d | features_daily | 同上 |
| 可选 avg_amount_20 | features_daily | 由 **amount** 算出 |
| 股票列表 | picks_daily(UNIVERSE) | 来自 ingest_universe |

**结论**：V1 只需 **prices_daily（close, amount）** + **universe**。二者当前均来自 akshare，无需其他渠道。

### 1.3 V2 / V3（趋势 + ML）

| 需求 | 来源 | 说明 |
|------|------|------|
| 趋势/风险特征 | features_daily | ret_10d, ret_20d, ma*, vol_ann_60d, mdd_60d, limit_down_60 等，均由 **close**（及 amount 用于流动性）在 build_features 中计算 |
| 单标的 ML 输入 | prices_daily | 每只股票的历史 **close, amount**，在 v2_v3_v4 里拉 420 日以内，做 LR+GBDT 特征（ret1/3/5/10, gap_ma*, vol_10/20, amount_chg） |
| 股票列表 | universe | 同上 |

**结论**：V2/V3 只需 **prices_daily（close, amount）** + **universe**。全部可由 akshare 行情满足。

### 1.4 V4（趋势质量 + 回测）

| 需求 | 来源 | 说明 |
|------|------|------|
| 与 V2/V3 相同 | features_daily + prices_daily | 同上 |
| bt_mean, bt_winrate 等 | features_daily | build_features 用 **close** 做滚动回测（LUMP/DCA5/DCA10）得到 |

**结论**：V4 同样只需 **prices_daily（close, amount）** + **universe**，无需新数据源。

### 1.5 V5（截面/池化模型，可选）

| 需求 | 来源 | 说明 |
|------|------|------|
| 数值特征 | features_daily | 与 V2–V4 相同 |
| **行业/板块** | 需额外接口 | 截面因子、行业分散、行业中性化 |
| **市值/流通市值** | 需额外接口 | 截面分层、流动性过滤、风格控制 |

**结论**：V5 若只做“全市场一个模型、仅用 features_daily”，则 akshare 行情足够；若要做**行业/市值**等截面因子，则需要**行业与市值**数据（见下节）。

## 2. 从 akshare 需要获取的数据（清单）

### 2.1 当前已用接口

| 接口 | 用途 | 产出字段/表 |
|------|------|-------------|
| `ak.stock_info_a_code_name()` | 股票列表 | 代码、名称 → universe，写入 picks_daily(UNIVERSE) + lake/universe |
| `ak.stock_zh_a_hist(symbol, period="daily", start_date, end_date, adjust="")` | 单只 A 股日频行情 | 日期、开高低收、成交量、成交额 → prices_daily（open, high, low, close, volume, amount） |
| `ak.tool_trade_date_hist_sina()` | 交易日历 | 交易日期列表 → TradingCalendar |

### 2.2 必须从 akshare 拿到的“最小集”

构建 V1–V4（及仅用价格特征的 V5）**最少**需要：

1. **Universe**  
   - 一次调用：`stock_info_a_code_name()`  
   - 得到：全 A 股代码 + 名称，用于确定标的池与 ticker 映射（如 000001 → 000001.SZ）。

2. **日频行情（每只标的）**  
   - 按标的、按日拉取：`stock_zh_a_hist(symbol, period="daily", start_date, end_date, adjust=?)`  
   - **必须字段**：  
     - **日期**（trade_date）  
     - **收盘价**（close）—— 用于收益、均线、波动、回测、ML 特征  
     - **成交额**（amount）—— 用于 avg_amount_20、流动性、ML 中 amount_chg  
   - **建议字段**：开盘、最高、最低（open, high, low）、成交量（volume），便于未来扩展（VWAP、波动率等）且 schema 已支持。

3. **交易日历**  
   - 一次/按需：`tool_trade_date_hist_sina()`  
   - 用于：确定 lookback 窗口（约 260 个交易日）、评估 horizon（1/5/10 日）、run_daily 日期推断。

### 2.3 复权（adjust）

- 当前代码中 `adjust=""` 为**不复权**。  
- **建议**：收益与回测统一用**前复权**（如 `adjust="qfq"`），避免分红送股导致收益计算失真。  
- 需在 `AkSharePriceProvider` 中改为可配置（如 config 中 `prices_adjust: "qfq"`），默认 `qfq`，兼容现有缓存策略（换复权需清缓存或换 key）。

### 2.4 可选：从 akshare 补充的接口（用于 V5 / 分析）

| 接口 | 可能字段 | 用途 |
|------|----------|------|
| `ak.stock_zh_a_spot_em()` | 现价、涨跌幅、成交额、**总市值、流通市值**等 | 当日截面：市值、流动性；项目内 hot_industry 已用 |
| `ak.stock_board_industry_name_em()` | 行业板块列表 | 行业列表，用于获取成分 |
| `ak.stock_board_industry_cons_em(symbol="行业名")` | 某行业成分股 | 标的–行业映射；同花顺另有 `stock_board_industry_cons_ths` 等 |
| `ak.stock_info_a_code_name()` | 已用 | 仅代码+名称，无行业/市值 |

行业/市值若只做“当日截面”，可用 `stock_zh_a_spot_em()` 一次拉全市场，再与 universe 用代码关联；若要做历史截面，需考虑 akshare 是否提供历史市值/历史行业成分，或引入其他源。

## 3. 需要从其他渠道获得的数据（及获取方式）

### 3.1 当前“必须实现”的模型（V1–V4）

- **不需要**其他数据源；**仅 akshare 的 universe + 日频行情 + 交易日历**即可。

### 3.2 若要做“带行业/市值”的 V5 或增强分析

| 数据 | 用途 | 可选来源 | 获取方式简述 |
|------|------|----------|--------------|
| **行业分类** | 截面因子、行业分散、行业中性 | akshare（东方财富/同花顺板块成分）、Tushare、Wind/聚宽 | akshare：先取行业列表，再按行业取成分，得到“标的–行业”；可每日/每周更新落表 |
| **市值（总市值/流通市值）** | 截面因子、流动性过滤、分层 | akshare `stock_zh_a_spot_em()`、Tushare、Wind | akshare：全市场 spot 一次请求含市值；历史需日频快照或 Tushare 等 |
| **基本面（PE/PB/ROE 等）** | 未来多因子/价值风格 | Tushare、akshare 部分接口、Wind | 按需接；频率多为季度/半年度，需对齐到日频（前推） |
| **停牌/涨跌停状态** | 过滤、解释 limit_up_60/limit_down_60 | 可从行情 open/close 近似，或 akshare 盘口 | 当前 limit_up_60/limit_down_60 已由收益≈±9.7% 近似，可暂不单独接 |

### 3.3 推荐优先级

1. **只做 V1–V4**：不接其他渠道，仅把 akshare 行情 + 复权 + 缓存/回填做好即可。  
2. **做 V5 且要行业/市值**：  
   - 先用 akshare：`stock_zh_a_spot_em()` 取当日市值；行业用 `stock_board_industry_cons_em` 等拼“标的–行业”表并定期更新。  
   - 若 akshare 历史行业/市值不全，再考虑 Tushare（需 token）或本地 Wind/聚宽导出的历史截面表。

## 4. 数据工程应如何组织

### 4.1 分层与职责（与现有一致，略作规范）

| 层 | 目录/表 | 内容 | 谁写 | 谁读 |
|----|----------|------|------|------|
| 原始/缓存 | data/lake/universe/, data/lake/prices/ | 按日或按标的的 Parquet，避免重复请求 | ingest_universe, ingest_prices | ingest_* 回填、失败回退 |
| 结构化存储 | data/store/alpha_tracker.duckdb | prices_daily, features_daily, picks_daily 等 | 各 pipeline 通过 DuckDBStore | build_features, score_all, eval, dashboard |
| 产出 | data/out/ | CSV 等给 dashboard/分析 | portfolio_nav, make_dashboard 等 | 人/Dashboard |

原则：**所有路径与库名来自 config**；**只通过 DuckDBStore 写 DuckDB**；大面板优先写 lake 再按需入 store。

### 4.2 采集（Ingestion）工作流

1. **Universe**  
   - 入口：ingest_universe（或 build_universe_hot 等）。  
   - 调用：akshare `stock_info_a_code_name()`。  
   - 输出：picks_daily(version='UNIVERSE')；可选写 lake/universe/YYYY-MM-DD/universe.parquet。  
   - 失败：可回退到 lake 中最近一次 universe 快照（已有逻辑）。

2. **Prices**  
   - 入口：ingest_prices。  
   - 输入：ticker 列表（来自 picks_daily UNIVERSE 或配置）、日期范围（由 trade_date + last_n 或 start/end 决定）。  
   - 调用：按 ticker 循环 `stock_zh_a_hist(..., adjust=config)`；建议**先读 config 的复权参数**。  
   - 缓存：按 (ticker, start, end) 写 lake/prices/{ticker}/{start}_{end}.parquet；请求失败时读缓存。  
   - 写入：按 (trade_date, ticker) 幂等写入 prices_daily（当前已用 DELETE 窗口再 INSERT）。

3. **交易日历**  
   - 在 TradingCalendar 中调用 `tool_trade_date_hist_sina()`；可增加“按区间缓存到本地”避免频繁请求（可选）。

### 4.3 窗口与回填（Backfill）

- **build_features** 需要约 **260 个交易日** 的行情（MA60、vol_ann_60d、mdd_60d、rolling backtest 约 140 日等）。  
- **ingest_prices** 的 lookback 需 ≥ 260 交易日（自然日约 420 天已用）；run_daily 的 `--last-n` 要覆盖该长度。  
- **首次/回填**：  
  - 先跑 ingest_universe（目标日或最近日）；  
  - 再对目标 ticker 列表跑 ingest_prices，start = 目标日 − 420 天（或按日历换算 260 交易日），end = 目标日；  
  - 再 build_features → score_all；  
  - 批量历史日期可用 backfill_runs 或脚本循环 start/end。

### 4.4 质量与一致性

- **缺失**：某日某 ticker 无行情时，不写该 (trade_date, ticker)；build_features 会因缺失日期导致该标的该日无特征或 NA，可打日志或记入 meta_runs。  
- **重复**：prices_daily 主键 (trade_date, ticker)，ingest 时先 DELETE 再 INSERT 窗口内数据，避免重复。  
- **复权一致性**：全库统一一种 adjust；更换时需清空或重建该段时间的 lake 与 prices_daily。  
- **数据新鲜度**：run_daily 每日拉取 last_n 天即可覆盖“当日特征计算”所需；若要做长区间回测，需对整段区间做一次 ingest_prices。

### 4.5 若增加行业/市值（为 V5 或分析）

- **表**：在 DuckDB 中可增 `stock_info_daily` 或 `stock_sector_daily`（trade_date, ticker, industry_name, total_mv, circ_mv 等）。  
- **采集**：  
  - 每日/每次 run_daily 前：调用 `stock_zh_a_spot_em()` 取当日市值；  
  - 行业：定期（如每周）拉取 `stock_board_industry_name_em()` + 各行业 `stock_board_industry_cons_em(symbol=...)`，生成 ticker–industry 映射表（可带 trade_date 或“当前有效”）。  
- **写入**：单独脚本或 ingest 子模块写 DuckDB；build_features 或 V5 读时 join 该表。

## 5. 实施清单（数据侧）

| 序号 | 事项 | 说明 |
|------|------|------|
| 1 | 确认 akshare 最小集 | universe + 日频行情(close, amount 必选；open/high/low/volume 建议) + 交易日历 |
| 2 | 复权可配置 | default.yaml 增加 `prices_adjust: "qfq"`；AkSharePriceProvider 使用该参数；缓存 key 含 adjust |
| 3 | 行情缓存 key | lake/prices 的路径或 cache key 包含 adjust，避免混用复权 |
| 4 | 回填与窗口 | 文档化“260 交易日 ≈ 420 自然日”；backfill 时 start 覆盖该长度 |
| 5 | （可选）行业/市值 | 新增 ingest_sectors 或 ingest_spot；写 stock_info_daily 或 sector 表；V5 再读 |

---

## 6. 小结

- **构建 V1–V4**：只需 akshare 的**股票列表**、**日频行情（至少 close + amount）**、**交易日历**；无需其他渠道。  
- **V5（仅用价格特征）**：同上。  
- **V5（带行业/市值）或后续多因子**：需要**行业**与**市值**，可先用 akshare 的 spot + 板块成分；历史不足时再考虑 Tushare 等。  
- **数据工程**：保持“lake 缓存 + store 结构化 + 只通过 DuckDBStore 写库”；复权可配置；窗口与回填按 260 交易日设计；行业/市值作为可选层按需加表和采集脚本。
